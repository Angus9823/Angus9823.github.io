✅ 配置目标 存储用户信息到 Redis； 使用 HSET 命令将用户名 angus 的密码哈希和 salt 存入 Redis； 供 EMQX 等 MQTT Broker 使用以下命令查询认证： bash 深色版本 HMGET mqtt_user:angus password_hash salt 🔐 认证方式说明 EMQX 在使用 Redis 做认证时，通常会配置如下规则：

bash 深色版本 auth.redis.hget_cmd = HMGET mqtt_user:${username} password_hash salt 这意味着：

EMQX 会使用用户名构造 Redis Key：mqtt_user:angus 然后使用 HMGET 获取 password_hash 和 salt； 再使用你客户端传来的密码 + salt 做加密，与 password_hash 比较。 🧱 Redis 存储结构 你将使用 Redis 的 Hash 类型 来存储用户信息：

bash 深色版本 HSET mqtt_user:angus password_hash “加密后的密码” salt “加密用的盐” ✅ 配置步骤

  1. 生成 salt（可选） 你可以随机生成一个 salt，例如：

bash 深色版本 openssl rand -hex 16 输出示例：

深色版本 a1b2c3d4e5f67890 你可以使用这个作为 salt。

  1. 生成 password_hash 你需要将 angus@2025 加上 salt 后做哈希加密（常见算法：SHA256、PBKDF2、bcrypt 等）。

示例：使用 SHA256 加密（简单示例） bash 深色版本 echo -n “angus@2025a1b2c3d4e5f67890” | openssl dgst -sha256 输出示例：

深色版本 (stdin)= 2bb80d537b14e17e7f551a7d3adf5ccc7864fcfb07aa04039c433280f28f9191 这就是你的 password_hash。

  1. 将用户信息写入 Redis 使用 redis-cli 插入用户信息：

bash 深色版本 redis-cli 然后执行：

bash 深色版本 HSET mqtt_user:angus password_hash “2bb80d537b14e17e7f551a7d3adf5ccc7864fcfb07aa04039c433280f28f9191” salt “a1b2c3d4e5f67890” 4. 测试是否写入成功 bash 深色版本 HMGET mqtt_user:angus password_hash salt 输出应为：

深色版本

properties 深色版本 auth.redis.server = 127.0.0.1:6379 auth.redis.password = your_redis_password_if_any auth.redis.pool_size = 8 auth.redis.database = 0 auth.redis.hget_cmd = HMGET mqtt_user:${username} password_hash salt auth.redis.password_hash = sha256 auth.redis.salt = a1b2c3d4e5f67890 ⚠️ 注意：如果你已经在 Redis 中写入了 salt，这里 auth.redis.salt 要设置为 undefined，否则 EMQX 会忽略 Redis 返回的 salt 字段。

MediaServer进程

这是ZLMediaKit作为服务器的主进程，该进程可以在免去开发的情况下直接作为测试流媒体服务器使用，如果你需要更复杂的业务逻辑，可以通过Web HOOK和RESTful API实现,同时你可以通过配置文件控制其参数。

  • 说明：
    • -d(–daemon): 是否以守护进程的方式启动，守护进程只做一件事，就是判断子进程(这个才是干活的进程)是否已经退出，退出后会不断尝试重启子进程。
    • -l(–level): 指定日志打印等级，赋值范围为0~4，等级越高，日志越少。
    • -m(–max_day): 日志文件保存天数，程序本次运行期间的日志如果超过这个天数，就会被删除。
    • -c(–config): 指定配置文件路径，配置文件为ini格式，请参考ZLMediaKit的默认配置文件。
    • -s(–ssl): 指定ssl证书路径，证书格式支持p12和pem类型，里面必须包含公钥和私钥，私钥不能有加密密码。如果指定文件夹，会加载文件夹下所有证书。
    • -t(–threads): 指定事件驱动线程(干重活)和后台工作线程(干阻塞的活)个数。
